# Flavorful — Day 3 Lab: Acquia Cloud, Composer, Drush, Config & Deployment

> Companion to the 5-day Drupal/Acquia vetting prep guide. This is the mandatory DevOps day. The single most valuable thing you can walk into the call able to do is **narrate a real deployment through environments**. Every step has a *why* attached — the vetting lead probes the reasoning, not just the commands.
>
> **Section 6 is grounded in a real Acquia Cloud Next deployment of this very project** (`docs/actual-outcomes/acquia-deployment-guide.md`), so it reflects what actually worked, not the idealised version.

**Build target:** Flavorful under Git + Composer; all Day 1–2 configuration exported to `config/sync` and committed; a contrib module added via Composer; a Config Split so a dev-only module is enabled only on dev; the repo pushed to GitHub with CI on pull requests; and (with a free trial) deployed once to an Acquia Cloud environment. By the end you can narrate the full release start to finish.

## 0. The mental model (read this first)

Enterprise Drupal on Acquia runs on three ideas:

- **Everything is code except content.** Structure — content types, fields, views, roles, settings — is *configuration*, exported to YAML in Git. The recipes and chefs editors create are *content*, living only in the database. You version and deploy the first; you copy the second.
- **Code moves up, data moves down.** Code (including config YAML) is deployed **up**: Dev → Stage → Prod. Databases and files are copied **down**: Prod → Stage → Dev, so lower environments hold realistic content to test against.
- **Never click-configure production.** You make structural changes locally, export them, commit, and deploy. Production config is applied by importing YAML — not by clicking around.

> **Interview line:** "Config is code and moves up through environments; content is data and gets copied down. I never make structural changes directly on production — I export config locally, commit it, and it's imported on deploy."

## 1. Put Flavorful under Git + Composer

**Why:** Drupal is a Composer-managed application. Composer controls exactly which versions of core and every contrib module you have, so any teammate or environment can rebuild an identical codebase from two files.

### 1.1 Initialise Git

```bash
cd <project-root>   # contains composer.json and the web/ (or docroot/) docroot
git init
git add .
git commit -m "Initial Flavorful codebase"
```

### 1.2 Understand (and set) .gitignore

You do **not** commit downloaded code (`vendor/`, core, contrib) — Composer regenerates it from `composer.json` + `composer.lock`. You commit the two Composer files, your custom code, and your config:

```gitignore
# Dependencies (rebuilt by `composer install`)
/vendor/
/web/core/
/web/modules/contrib/
/web/themes/contrib/
/web/libraries/

# Environment / local
/web/sites/*/settings.local.php
/web/sites/*/files/
.env
```

> **Acquia nuance:** locally you gitignore `vendor/` and core/contrib. For deployment, Acquia expects a **build artifact** that *does* include `vendor/` and core — your pipeline (or `acli push:artifact`) runs `composer install` and pushes the built artifact to Acquia's Git. "What you commit" and "what you deploy" differ, and the pipeline bridges them. (See Section 6.)

### 1.3 Add a contributed module the right way

The daily loop: require → enable → configure → export config → commit. We add **Pathauto** (automatic clean URLs like `/recipes/spaghetti-bolognese`) because it genuinely helps the recipe site's SEO.

```bash
composer require drupal/pathauto      # updates composer.json + composer.lock (commit both)
drush en pathauto -y
drush cr
```

Configure a URL pattern at `/admin/config/search/path/patterns` → Add pattern → for **Recipe** use `recipes/[node:title]`. Then export the config (Section 3) and commit it — the pattern is configuration, so it travels to other environments.

### 1.4 Two files you must be able to explain

| File | What it is | Commit it? |
|---|---|---|
| `composer.json` | Your declared, intended dependencies and version constraints (e.g. `^11`). | Yes |
| `composer.lock` | The exact resolved versions of every package — the reproducible snapshot. | Yes |
| `vendor/` | The actual downloaded code, generated by `composer install` from the lock file. | No (gitignored) |

> **Gotcha:** never edit anything inside `vendor/` or `web/core/` by hand — the next `composer install` wipes it. To change a contrib module, use a **patch** via `cweagans/composer-patches` (declared under `extra.patches` in `composer.json`), so the fix is reproducible.

### 1.5 Push to a remote so you can open a PR and run CI

**Why:** a pull request and CI only exist once your code lives on a shared Git host. Section 1 only did a local `git init` — this is the prerequisite for Sections 5 and 6.

```bash
# 1. Create an empty repo on GitHub (New repository -> "flavorful", no README).
git remote add origin git@github.com:<your-user>/flavorful.git
git branch -M main
git push -u origin main

# 2. Feature branch for your next change
git checkout -b feature/nutrition-field
git add .
git commit -m "Add total-calories field + exported config"
git push -u origin feature/nutrition-field
```

On GitHub click **Compare & pull request** → open the PR against `main`. This is the review surface: teammates read the code diff *and* the `config/sync` YAML diff. CI (Section 5) runs automatically on this PR.

> **What a PR actually is:** a request to merge one branch into another, with a diff and a comment thread. Nothing deploys when you open it — it's a review gate.

## 2. Drush — the CLI you'll use every day

**Why:** Drush is the command-line companion to Drupal. You're expected to reach for it reflexively.

| Command | What it does / when |
|---|---|
| `drush cr` | Rebuild cache. The universal "why isn't my change showing?" fix. |
| `drush cex` | Export active config from the DB to `config/sync` YAML. |
| `drush cim` | Import config from YAML into the site (applies a release's structural changes). |
| `drush cst` | Config status — diff between the DB and the YAML files. |
| `drush updb` | Run pending database updates (after core/module version changes). |
| `drush en / pmu` | Enable / uninstall a module. |
| `drush uli` | Generate a one-time admin login link. |
| `drush sql:dump` | Export the database. |
| `drush ws` | Tail the logs (watchdog). |

## 3. Configuration management (the heart of Day 3)

**Why:** this is the mechanism that lets a team move a content type or view from a laptop to production safely and reviewably.

### 3.1 Set the config sync directory

In `web/sites/default/settings.php`:

```php
$settings['config_sync_directory'] = '../config/sync';
```

That puts YAML in a `config/sync` folder **outside** the docroot (not web-accessible) but inside your Git repo.

### 3.2 Export, inspect, commit

```bash
drush cex -y
```

Look inside `config/sync` — human-readable YAML like `node.type.recipe.yml`, `field.field.node.recipe.field_cuisine.yml`, `views.view.recipes.yml`. Every structural thing you built on Days 1–2 is now a file. Commit it:

```bash
git add config/sync && git commit -m "Export Flavorful config"
```

### 3.3 See the workflow in action

1. Make a structural change — add a field, e.g. `field_total_calories` on Recipe.
2. Run `drush cst` — it reports config in the DB differs from the files.
3. Run `drush cex -y` and inspect `git diff` — a new field YAML file. **This diff is what a colleague reviews in your PR.**
4. To simulate a deploy target, pull the change on another environment (or a fresh DB) and run `drush cim -y` — the field is created from YAML, no clicking.

> **The #1 ordering gotcha:** on deploy always run `drush updb` **before** `drush cim`. Module updates can change config *schema*; running DB updates first prevents `cim` failing on a schema mismatch. Canonical post-deploy order: `updb → cim → cr`.

### 3.4 Environment-specific config: Config Split

**Why:** some config should differ per environment — e.g. the `Devel` module on dev but never prod; Twig debug on locally, off live.

```bash
composer require drupal/config_split
drush en config_split -y
```

At `/admin/config/development/configuration/config-split` create a split named **Development**, storage folder `config/splits/dev`, and mark `Devel` as part of it. Activate it only on dev via a settings override:

```php
$config['config_split.config_split.dev']['status'] = TRUE;   // dev settings only; FALSE on prod
```

### 3.5 Secrets and per-environment values

API keys, DB credentials, and environment URLs must **never** be committed as config. Keep them in `settings.php` overrides / environment variables (on Acquia, injected per environment). Your Day 5 nutrition-API key belongs here, not in `config/sync`.

## 4. Acquia Cloud Platform (the model)

**Why:** the JD requires "strong experience with Acquia Cloud." Here's the mental map; the hands-on is in Section 6.

### 4.1 Environments & how things move

| | Dev | Stage | Prod |
|---|---|---|---|
| Purpose | integrate features | UAT / client review | live site |
| Gets code | first (feature branches) | release candidate | tagged release |
| Gets data | copied down from Prod | copied down from Prod | source of truth |

Code moves **up** by deploying a branch/tag; databases and files are dragged **down** from Prod to refresh Stage/Dev.

### 4.2 Cloud Hooks — automating the post-deploy steps

**Why:** you don't want to SSH in and run Drush by hand on every release. **Cloud Hooks** are scripts committed in a `hooks/` directory that Acquia runs automatically on events like `post-code-deploy`:

```sh
#!/bin/sh
# hooks/common/post-code-deploy/deploy.sh
drush @$site.$target_env updb -y
drush @$site.$target_env cim -y
drush @$site.$target_env cohesion:import   # Site Studio assets
drush @$site.$target_env cohesion:rebuild  # Site Studio rebuild
drush @$site.$target_env cr
```

> On Acquia Cloud Next this project ran these steps via `code-switch` + remote Drush (Section 6); Cloud Hooks remain the way to automate them.

### 4.3 The rest of the toolbox (know what each is for)

- **Acquia CLI (`acli`):** command-line control of Cloud — build/push artifacts, deploy, run remote Drush, pull a DB down.
- **Drush site aliases:** shorthand for running Drush against a remote environment, e.g. `drush @flavorful.prod cr`.
- **Acquia Cloud IDE:** browser-based dev environment pre-configured for Drupal/Acquia.
- **Acquia Cloud Site Factory (ACSF):** runs many sites from one codebase (multisite at scale). Know what it's for.
- **BLT is end-of-life** (archived early 2025; never supported Drupal 11). Replaced by **Drupal Recommended Settings (DRS)** plus Composer/CI automation. Saying this shows you're current.

## 5. Continuous Integration — actually run it

**Why:** CI automatically builds and tests your code every time you push or open a PR, catching mistakes before they reach an environment. We'll wire real CI with **GitHub Actions** (free).

### 5.1 What happens the moment you open the PR

- Pushing the branch / opening the PR **triggers your CI workflow automatically**.
- CI spins up a clean container, installs dependencies, and runs your checks.
- **Green** = safe to review and merge. **Red** = fix and push again; the PR re-runs CI on each push.
- After approval you **merge to main** — and merging triggers the deploy step (5.4 + Section 6).

### 5.2 Add a GitHub Actions workflow

Create `.github/workflows/ci.yml` (GitHub auto-detects anything under `.github/workflows/`):

```yaml
name: CI
on:
  pull_request:
  push:
    branches: [ main ]
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.3'
          tools: composer
      - name: Install dependencies
        run: composer install --no-interaction --no-progress
      - name: Coding standards (Drupal)
        run: vendor/bin/phpcs --standard=Drupal,DrupalPractice web/modules/custom web/themes/custom
      - name: Static analysis
        run: vendor/bin/phpstan analyse web/modules/custom
      - name: Tests
        run: vendor/bin/phpunit web/modules/custom || true
```

> **First-time setup:** the check tools are dev dependencies — add them once: `composer require --dev drupal/coder phpstan/phpstan mglaman/phpstan-drupal`. The `|| true` on tests is a starter crutch so an empty test suite doesn't fail the build — remove it once you write real tests.

### 5.3 Run the exact same checks locally first

```bash
composer install
vendor/bin/phpcs --standard=Drupal,DrupalPractice web/modules/custom web/themes/custom
vendor/bin/phpstan analyse web/modules/custom
```

Being able to say "CI just runs the same phpcs/phpstan/phpunit I run locally" is exactly the level of understanding the vetting call wants.

### 5.4 From green CI to a deploy (the CD half)

**Continuous Delivery** adds a deploy step after tests pass on `main`. On Acquia the deployable thing is a **build artifact** (code + `vendor/`) pushed to Acquia's Git and applied to an environment:

- **Manually** — merge, then deploy in the Acquia UI / via `acli` (Section 6). Legitimate, and how many teams run.
- **Automatically** — a workflow step (or Acquia Code Studio pipeline) runs `acli push:artifact` on merge to `main`, then activates and runs the Drush sequence.

> **For your prep:** wiring the CI half (5.2) on a free GitHub repo is enough to *show* you've done CI/CD. The auto-deploy half needs the Acquia credentials from Section 6.

## 6. Push Flavorful to Acquia Cloud (hands-on) — what actually worked

> Rewritten from a real **Acquia Cloud Next** deployment of a Composer-managed Drupal 11 site (developed locally with DDEV). Every step and gotcha is what *actually worked*. Full runbook: [`../actual-outcomes/acquia-deployment-guide.md`](../actual-outcomes/acquia-deployment-guide.md).

### 6.1 The one idea that changes everything

**Acquia is NOT a "git push and it runs" host.** Two facts drive the whole process:

1. Acquia serves **exactly what is committed to the deploy branch** — there is **no** `composer install` on deploy, so `vendor/`, core and contrib must physically be in that branch. Since those are gitignored, you build a **deployment artifact** (a throwaway commit containing source + installed dependencies) with `acli`.
2. The document root is fixed at `/var/www/html/docroot` — your repo must serve from `docroot/`, not `web/`.

### 6.2 Prerequisites

- **Acquia CLI (`acli`):** download `acli.phar` from GitHub releases into `/usr/local/bin/acli`.
- **Host Composer:** `brew install composer` — `acli` shells out to a *real* Composer on your machine, not `ddev composer`.
- **A new-enough host PHP:** Composer loads plugins at runtime, and Drupal 11 contrib uses PHP 8.3 syntax — an older host PHP fatals with a `ParseError` before doing anything. Build under 8.3+ (this project used 8.5).
- **Acquia account** with team access to the target application, and the app's **UUID**.

### 6.3 SSH key — it must be RSA (not ed25519)

> **Correction worth remembering:** Acquia's Git/SSH endpoints **reject ed25519 keys** with `the supplied public key type is unsupported: it must be RSA`.

```bash
ssh-keygen -t rsa -b 4096 -C "you@example.com (acquia)" -f ~/.ssh/acquia_key -N ""
cat ~/.ssh/acquia_key.pub   # paste into Acquia UI: Profile -> SSH keys -> Add SSH key
```

Point Acquia hosts at this key in `~/.ssh/config`. If you use a 1Password / agent `Host *` block it will answer first (→ `Permission denied`) — override it here, and put this block **before** any `Host *`:

```ssh-config
Host *.hosting.acquia.com *.acquia-sites.com   # Git + per-env SSH domains
  IdentityFile ~/.ssh/acquia_key
  IdentitiesOnly yes
  IdentityAgent none          # bypass any agent from a later Host * block
```

Verify (allow 1–10 min to propagate): `ssh -T <git-remote-host>` → "Welcome to Acquia Cloud" = success. `ssh -vT` shows which key is offered if denied.

### 6.4 Authenticate acli & discover the environment

```bash
acli auth:login                                  # uses a Cloud API token (Account -> API tokens)
acli api:accounts:find                           # confirm you're the right user
acli api:applications:environment-list <APP_UUID>
```

Record the **Environment ID**, the **deploy branch** (from the env's `vcs.path` — this project's was `dist`), the **Git remote**, the **env SSH** string, and the **public URL**.

### 6.5 Prepare the codebase (three fixes that each cost a debugging cycle)

**(a) Pin the PHP platform** so the artifact resolves against Acquia's runtime (8.3):

```jsonc
// composer.json
"config": { "platform": { "php": "8.3.0" } }
```

Refresh the lock hash where you have PHP 8.3 (inside DDEV): `ddev composer update --lock --no-install`.

**(b) Add an Acquia-aware `settings.php` — and un-ignore it.** `sites/default/` is read-only on Acquia and Acquia injects DB credentials via a generated include. One committed file serves Acquia/DDEV/local:

```php
// docroot/sites/default/settings.php
if (file_exists('/var/www/site-php') && isset($_ENV['AH_SITE_GROUP'])) {
  require '/var/www/site-php/' . $_ENV['AH_SITE_GROUP']
        . '/' . $_ENV['AH_SITE_GROUP'] . '-settings.inc';   // Acquia
}
if (getenv('IS_DDEV_PROJECT') == 'true' && file_exists(__DIR__ . '/settings.ddev.php')) {
  include __DIR__ . '/settings.ddev.php';                    // DDEV
}
$settings['config_sync_directory'] = '../config/sync';
```

> **The gitignore trap:** `acli push:artifact` honors `.gitignore` when mirroring source into the artifact, and the default Drupal `.gitignore` ignores `sites/*/settings.php` — so even a committed `settings.php` gets filtered out and the deployed site has no settings. **Remove that ignore line** (keep `settings.local.php` / `settings.ddev.php` / `services.yml` ignored). This file holds no secrets — real credentials live in the Acquia include.

**(c) Match the document root: `web/` → `docroot/`.** Acquia serves `docroot/`. If your project uses `web/`, the homepage 200s to Acquia's "Welcome to Acquia Cloud" placeholder and every Drupal path 404s. Relocate:

```bash
# 1. composer.json: web-root and installer-paths  web/ -> docroot/
# 2. git mv web docroot
# 3. update .gitignore (web/ -> docroot/), .ddev/config.yaml (docroot: docroot), CI paths
ddev composer install            # re-scaffolds core/contrib under docroot/
ddev composer update --lock --no-install
```

Confirm the target on the server: `ssh <ENV_SSH> 'echo $DOCROOT'` (or `ls /var/www/html/`).

### 6.6 Build & push the artifact

`acli` needs a **clean working tree** (commit or `git stash --include-untracked` first). Build under a PHP that can parse your deps, with memory unlocked:

```bash
PATH="/opt/homebrew/opt/php/bin:$PATH" \
  php -d memory_limit=-1 /usr/local/bin/acli push:artifact <ENV_ID> \
    --destination-git-branch=dist \
    -u <ACQUIA_GIT_URL> \
    --no-interaction
```

It clones `dist` → compiles the project in a temp dir (`composer install --no-dev`) → copies in `vendor/` → commits → pushes to `dist`. Success looks like `… dist -> dist  ✔`.

### 6.7 Activate it — the deploy is two-phase

> **Push ≠ deploy:** pushing updates Git but does **not** make the code live. Trigger a **code-switch** to activate the branch, then poll until complete.

```bash
acli api:environments:code-switch <ENV_ID> dist
# returns a notification UUID; poll it until "completed":
acli api:notifications:find <NOTIFICATION_UUID> | grep -E '"status"|"label"'
```

### 6.8 Load a database & finalise

```bash
ddev export-db --file=/tmp/local-db.sql.gz
ssh <ENV_SSH> 'cd /var/www/html && vendor/bin/drush sql:drop -y'
gunzip -c /tmp/local-db.sql.gz | ssh <ENV_SSH> 'cd /var/www/html && vendor/bin/drush sql:cli'
ssh <ENV_SSH> 'cd /var/www/html && vendor/bin/drush updatedb -y && vendor/bin/drush cr'
# Fresh, content-free alternative:  drush site:install --existing-config
```

### 6.9 Clear Varnish & verify

Acquia fronts the site with Varnish, so any 404s/placeholder pages served *while it was broken* get cached (you'll see `X-Cache: HIT` with a high `Age` on a stale 404). Purge, then verify:

```bash
acli api:environments:domains-clear-varnish <ENV_ID> <DOMAIN>
curl -sSI http://<DOMAIN>/user/login | grep -iE 'HTTP/|x-cache|age'   # expect 200; MISS = fresh from Drupal
```

### 6.10 The gotchas we actually hit (great interview war-stories)

| Symptom | Root cause | Fix |
|---|---|---|
| `public key ... must be RSA` | Acquia rejects ed25519 | Use `ssh-keygen -t rsa -b 4096` |
| `Permission denied (publickey)` | 1Password `Host *` agent answers first | `Host` block, `IdentityAgent none`, both domains |
| `ParseError` in a Composer plugin | Host PHP too old to parse 8.3 deps | Build under PHP 8.3+ (they used 8.5) |
| `Allowed memory size exhausted` | 128M too small for full vendor scan | `php -d memory_limit=-1 acli …` |
| Homepage = "Welcome to Acquia", all 404 | App in `web/`; Acquia serves `docroot/` | Relocate web root to `docroot/` |
| `settings.php` missing after deploy | `acli` honors `.gitignore` | Un-ignore `settings.php` |
| Pushed to dist but site unchanged | Push ≠ deploy | `acli api:environments:code-switch` |
| `/user/login` still 404 after fixes | Stale Varnish cache of the old 404 | Clear Varnish for the domain |

> **Why this wins the call:** anyone can recite "build an artifact and deploy." Being able to say *"Acquia needs RSA keys, serves only what's committed so you push an artifact with vendor bundled, deploy is a two-phase code-switch, and watch Varnish when something looks stuck"* tells the lead you've **actually shipped to Acquia** and debugged it.

## 7. The money answer: narrate a full release

Rehearse this out loud until it's smooth — it covers Git, Composer, Drush, config, Acquia and CI/CD in one breath.

1. Cut a **feature branch** and build locally: `composer require` any new modules, build the feature, `drush cex` to capture config.
2. Open a **pull request**. Reviewers read the code and the `config/sync` YAML diff; CI runs coding standards, PHPStan, and tests.
3. Merge. CI **builds the artifact** (`composer install --no-dev` + front-end build, `vendor/` bundled) and `acli push:artifact` sends it to Acquia's deploy branch.
4. **Activate** it on Stage with a `code-switch` (push ≠ deploy), then run `updb → cim → (cohesion:import + rebuild) → cr` via a Cloud Hook or remote Drush. Stakeholders run **UAT** on Stage (refreshed with a Prod data copy).
5. Once signed off, **tag a release** and repeat the artifact push + code-switch to **Prod**; clear Varnish, smoke-test, done.

> **Why it lands:** it shows you understand that a release is code + config together, gated by review and tests, applied identically to every environment by automation — not by hand.

## 8. End-of-day verification (say these out loud)

- Explain **code up / data down** and why you never configure prod directly.
- `composer.json` vs `composer.lock` vs `vendor/` — what each is and which you commit.
- The config workflow: `cex` → commit → deploy → `cim`, and **why `updb` runs before `cim`**.
- What **Config Split** solves and a concrete example (Devel on dev only).
- What a **Cloud Hook** is and the exact sequence a post-deploy hook runs.
- Why **BLT** is no longer used and what replaced it.
- What happens **the instant you open a PR** (CI triggers) and what phpcs / phpstan / phpunit each check.
- How code reaches Acquia: build an **artifact** (vendor bundled, since Acquia runs no `composer install`) → **push** to the deploy branch → **code-switch** to activate → Drush + clear Varnish.
- Name three **real Acquia gotchas**: RSA keys only (not ed25519), the fixed `docroot/` root, and the `settings.php` gitignore trap.
- Deliver the **full release narrative** (Section 7) in under 90 seconds.

## 9. Quick-reference command sheet

```bash
# Composer
composer require drupal/MODULE          # add a module (commit .json + .lock)
composer update drupal/MODULE -W        # update with dependencies
composer install                        # rebuild vendor/ from the lock file

# Drush
drush cex -y / drush cim -y             # export / import config
drush cst                               # config diff (DB vs files)
drush updb -y                           # run DB updates (BEFORE cim on deploy)
drush cr                                # rebuild cache
drush uli                               # one-time admin login link

# Git / GitHub (PR + CI)
git remote add origin <github-url>
git push -u origin main                 # push; open PR on GitHub -> CI runs
vendor/bin/phpcs / phpstan / phpunit    # the same checks CI runs

# Acquia (what actually worked)
ssh-keygen -t rsa -b 4096 -f ~/.ssh/acquia_key    # RSA only, NOT ed25519
acli auth:login                                   # Cloud API token
acli api:applications:environment-list <APP_UUID> # find ENV_ID, deploy branch, git URL

  acli push:artifact <ENV_ID> --destination-git-branch=dist -u <GIT_URL> --no-interaction

updb  ->  cim  ->  cohesion:import  ->  cohesion:rebuild  ->  cr
```
